# Security Policy

myClick processes the most sensitive data category there is — the biometrics of children. We take security disclosures seriously, and we'd far rather hear about a problem from you than discover it as an incident.

## Reporting a vulnerability

Email **security@myclick.camera**. Do not open a public GitHub issue for security matters.

Please include:

- A description of the vulnerability and where it lives
- Steps to reproduce, if you have them
- The potential impact as you see it

## What's in scope

- The iOS application
- The server-side end-to-end-encryption module
- The cryptographic protocol specification
- The data-access layer (the enumerated SECURITY DEFINER RPCs)

## What's out of scope

- The marketing site (report anything serious anyway, but it holds no sensitive data)
- Social-engineering or physical attacks against people
- Denial-of-service against our infrastructure

## Our commitment

- We'll acknowledge your report within 3 business days.
- We'll keep you updated as we investigate.
- We'll credit you publicly once a fix ships, if you'd like the credit.
- We won't pursue legal action against good-faith security research that respects this policy.

## Our trust posture

myClick's privacy claims are open-source so they can be verified, not trusted. If you find a place where the code doesn't match the claims we make in public, that is exactly the kind of disclosure we want. A paid third-party audit is committed for when we reach sustainable revenue; until then, the open code plus a responsive disclosure process are the trust mechanism.
